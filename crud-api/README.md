# CRUD API — Project Management

REST API for managing users, projects, tasks, and tags, built with Spring Boot 4 and PostgreSQL.

This project demonstrates a layered architecture with non-trivial JPA relationships (1-N, N-N), a complete DTO strategy, declarative validation, and a comprehensive integration test suite running against real PostgreSQL via Testcontainers.

## Overview

This REST API manages four related resources:

- **Users** — platform users
- **Projects** — projects owned by users
- **Tasks** — tasks belonging to projects
- **Tags** — labels applied to tasks

The project implements full CRUD operations on each resource with appropriate HTTP semantics, input validation, and clean serialization (no `LazyInitializationException` thanks to consistent DTO usage).

## Domain Model

### Entities and relationships

```
User (1) ────< (N) Project (1) ────< (N) Task (N) >──── (N) Tag
```

- `User → Project`: **One-to-Many** (a user owns several projects)
- `Project → Task`: **One-to-Many** (a project contains several tasks)
- `Task ↔ Tag`: **Many-to-Many** (a task can have multiple tags, a tag can be shared across tasks, through the `task_tags` join table)

### Identifier strategy

Each entity exposes two distinct identifiers:

- `id` (BIGSERIAL) — internal primary key, used for SQL joins and foreign keys. **Never exposed via the API.**
- `public_id` (UUID) — public identifier exposed in URLs and JSON payloads.

This pattern, inspired by production APIs like Stripe and Slack, provides several benefits:

- Joins remain fast (BIGINT index)
- Public IDs prevent enumeration attacks
- Internal IDs never leak to the outside

### DTO layer

Each endpoint uses dedicated DTOs (Java Records):

- `*CreateRequest` — for POST operations
- `*UpdateRequest` — for PUT operations
- `*Response` — for outgoing serialization

No JPA entity is ever serialized directly, eliminating any risk of `LazyInitializationException`.

### Validation

Jakarta Bean Validation (`@NotBlank`, `@Email`, `@Size`, etc.) is applied on all input DTOs. Validation failures return a structured `400 Bad Request` response with per-field error details.

## Getting Started

### Prerequisites

- Java 21
- Maven 3.9+ (wrapper included, no global install needed)
- Docker (for PostgreSQL and integration tests)

### 1. Start PostgreSQL

```bash
docker run -d \
  --name postgres-crud \
  -e POSTGRES_DB=crud_api \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 \
  postgres:17-alpine
```

### 2. Run the application

```bash
./mvnw spring-boot:run
```

The API is now available at: `http://localhost:8080/api`.

### 3. Build a runnable JAR

```bash
./mvnw clean package
java -jar target/crud-api-0.0.1-SNAPSHOT.jar
```

### Database schema

The schema is auto-generated by Hibernate (`spring.jpa.hibernate.ddl-auto=update`).
Tables created: `users`, `projects`, `tasks`, `tags`, `task_tags`.

## Configuration

Main configuration lives in `src/main/resources/application.yaml`:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/crud_api
    username: postgres
    password: postgres
  jpa:
    hibernate:
      ddl-auto: update
    open-in-view: false

server:
  port: 8080
  servlet:
    context-path: /api
```

> The `context-path: /api` means every route is prefixed with `/api`.
> Example: `GET /api/users`, `POST /api/projects`, etc.

```json
{
  "timestamp": "2026-05-22T10:00:00Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Email already in use: raymond@it-num.com",
```

| HTTP Code | Situation |
|---|---|
| `400 Bad Request` | Invalid body, failed validation constraints, business uniqueness conflict |
| `404 Not Found` | Non-existent resource (unknown publicId) |
| `500 Internal Server Error` | Unhandled server error |

## Integration Tests

The integration test suite uses **Testcontainers** to spin up an ephemeral PostgreSQL container for every test run. No pre-existing database is required — Docker just needs to be running on the machine.

### Test infrastructure

The test layer relies on three files at the root of `src/test/java/com/it_num/crud_api/`:

| File | Role |
|---|---|
| `CrudApiApplicationTests.java` | Spring context smoke test |
| `TestcontainersConfiguration.java` | Declares the `PostgreSQLContainer<?>` bean wired via `@ServiceConnection` |
| `TestCrudApiApplication.java` | Alternative `main` for running the app locally on a Testcontainers-backed database |

`TestcontainersConfiguration` exposes a single bean:

```java
@TestConfiguration(proxyBeanMethods = false)
public class TestcontainersConfiguration {
    @Bean
    @ServiceConnection
    PostgreSQLContainer<?> postgresContainer() {
        return new PostgreSQLContainer<>(DockerImageName.parse("postgres:17-alpine"));
    }
}
```

The `@ServiceConnection` annotation tells Spring Boot to automatically configure the JDBC URL, username, and password from the running container — no `application-test.yaml` override is needed.

### Test startup process

When running `./mvnw test`, the following sequence happens in the background:

1. **Maven Surefire forks a JVM** for tests, injecting the Docker API version via system properties (see [technical note](#technical-note-docker--testcontainers-compatibility))
2. **Testcontainers connects to the Docker daemon** through `/var/run/docker.sock`
3. **The `ryuk` companion container starts first** — it monitors the test JVM and automatically removes orphaned containers if the test process crashes
4. **A `postgres:17-alpine` container is pulled** (only on first run, cached afterwards)
5. **PostgreSQL starts on a random host port** (e.g. `localhost:32773`)
6. **Spring Boot reads the container's connection details via `@ServiceConnection`** and wires the JDBC `DataSource`
7. **Hibernate creates the schema** (`ddl-auto=create-drop` by default in tests): tables, indexes, foreign keys
8. **JUnit runs each test method** inside a `@Transactional` boundary, rolled back at the end for isolation
9. **At JVM shutdown, ryuk removes the test containers** — no manual cleanup needed

Each test class is isolated. Reusing the same Spring context across classes (the default Spring Boot behavior) means the PostgreSQL container is reused between test classes within a single Maven run — keeping execution time low.

## Technical Note: Docker / Testcontainers Compatibility

This project uses Spring Boot 4.0.6, which ships with Testcontainers 1.21.3 and docker-java 3.4.2. This combination has an incompatibility with Docker Engine 29.x: docker-java hardcodes the client API version `1.32`, while Docker 29.x requires a minimum of `1.44`. The symptom at test startup is:

> `client version 1.32 is too old. Minimum supported API version is 1.44`

The fix consists of exposing the API version through a Java system property that docker-java reads (environment variables are not picked up by this code path). The `pom.xml` configures `maven-surefire-plugin` accordingly:

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <configuration>
        <systemPropertyVariables>
            <api.version>1.44</api.version>
            <docker.api.version>1.44</docker.api.version>
        </systemPropertyVariables>
    </configuration>
</plugin>
```

This configuration will be removed once docker-java 3.5+ is integrated into a future Spring Boot release.

## Design Decisions

### Why `Set` instead of `List` for `@ManyToMany`?

`Set` prevents duplicate associations and avoids the `MultipleBagFetchException` when using `JOIN FETCH` on multiple collections.

### Why both `id` (Long) and `publicId` (UUID)?

- `id` (BIGSERIAL) is used for joins → fast index, small footprint
- `publicId` (UUID) is the only identifier exposed via the API → prevents enumeration attacks (`/users/1`, `/users/2` would be predictable)

### Why `@EntityGraph` on `TaskRepository`?

To prevent N+1 queries when fetching tasks with their project and tags. A single SQL query loads everything via JOINs.

### Why `open-in-view: false`?

Disabling Open Session In View forces explicit transaction management. With OSIV enabled, lazy-loading errors only show up in production. With OSIV disabled, they show up in tests, where they should be caught.

### Why separate `*CreateRequest` and `*UpdateRequest` DTOs?

- Different constraints can be enforced (e.g. `projectId` is required on create but immutable on update)
- Clearer intent in controller signatures
- Easier to evolve independently

### Why no `@Data` on entities?

`@Data` generates `equals`, `hashCode`, and `toString` over **all fields**, including LAZY relationships. This causes `LazyInitializationException` and infinite loops on bidirectional relationships. We explicitly use `@EqualsAndHashCode(of = "id")` and `@ToString(of = {...})` to control what's included.
